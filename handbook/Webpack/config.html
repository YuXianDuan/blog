    为什么要使用webpack

    # 为什么要使用webpack

    1. 开发时 使用的 一些框架(react vue) ES6 模块化语法 less sass等css预处理工具进行开发时 这样的代码想要在浏览器运行 必须编译成浏览器能识别的 js css 等语法 才能运行
    2. 压缩代码 兼容处理 提升代码性能

    # webpack是什么？

    # webpack本身的功能非常有限 webpack是一个静态资源打包工具

    1.开发模式：仅能编译js中的 ES Module 语法 像箭头函数 ，...运算符则不能编译需要借助插件 （mode=development）

    2.生产模式：能编译 js中的 ES Module语法 还能压缩代码体积(mode=production)

    ###webpack 下载

    安装webpac k npm i webpack webpack-cli -D 开发依赖

    打包 npx webpack ./src/main.js --mode=development

    # 处理样式资源

    1.借助loader来编译样式资源

    1.[style-loader,css-loader]处理css资源

    2.[style-loader,css-loader,sass-loader] 处理sass资源

    例子：

    module.exports ={
    module:{
        rules:[
            {
                //sass 和scss为后缀的文件
                test: /.s[ac]ss$/i,
                use:[
                    style-loader,//将 js模块编译成一个style标签插入到html中
                    css-loader,//将css文件编译成一个 common js模块 到js中
                    sass-loader//将 sass编译成 css文件
                ] 
            }
        ]
    }
}

    # 处理图片资源

    1.webpack4时对于图片资源使用 file-loader 和 url-loader处理

    file-loader将图片资源变成 webpack可以识别的资源

    url-loader将小于一定大小的图片转换成base64的格式

    图片转化成base64后图片加载不需要额外发请求了，但是图片资源的大小会变大

    2.webpack5已经将这两个loader 内置到webpack的内置功能中了，只需要通过配置去开启这两个功能 不需要loader 也就是说webpack本身已经支持这两个功能了

    例子

    module.exports={
    module:{
        rules:[
            {
                test:/.(png|gif|svg|jpe?g|webp)$/i,
                type:'asset',
                parser:{
                   dataUrlCondition:{
                       maxSize:20*1024,
                   }
                }
            }
        ]
    }
}

    # 处理字体视频等静态资源 不需要进行base64 处理

    针对这个问题 我们使用 babel来完成 高级语法转Es5 语法

    # css资源打包

    1. 开发模式下 css资源会通过 [css-loader,style-loader]打包成js资源，在去创建style标签 所以当js加载完后 才会给html插入style标签，加入样式

      所以当用户网速比较慢的时候它会出现闪屏显现

    2. 正确处理方式: 单独打包 css 文件 通过link标签加载性能才好

    3. 下载插件:npm install --save-dev mini-css-extract-plugin 解决闪屏问题

      # webpack优化

      # 提升开发体验

      1. SourceMap(源代码映射) 是一个用来生成源代码与构建后代码一一映射的文件方案， 它会生成一个 xxx.map文件从构建后代码的出错位置找到映射后源代码出错的位置，从而让浏览器找到映射后源代码出错的位置，帮助我们找到错误根源

      2. 开发模式：cheap-module-source-map

        1. 优点：打包编译速度快,只包含行映射
        2. 缺点： 没有映射列
        3. module.export={
    mode:"development",
    devtool:"cheap-module-source-map"
}

      3. 生产模式： source-map

        1. 优点：包含行列映射
        2. 缺点：打包编译速度更慢
        3. module.export={
    mode:"production",
    devtool:"source-map"
}

      4. 为什么开发模式要使用 cheap-module-source-map，生产模式要使用 source-map ，因为开发模式下代码不会进行压缩只需要映射到对应的行，生产模式代码只有一行 ，不加上列映射找不到错误

      # 提升打包速度

      1. HotModuleReplacement(HMR/热模块替换):在程序运行中 替换，添加，删除模块，无需重新加载整个页面 。就是不会重新打包整个项目，然后更新，只会更改 更改后文件对应的模块。

      2. webpack不支持js的热模块更新，只支持 css sass less 等样式资源的热更新 原因与 浏览器的事件循环有关

      3. 给 js添加热模块替换 原始方案

        if(module.hot){
    //判断浏览器是否支持热更新模块
    module.hot.accept('./js/count',
                      //副作用函数
                      ()=>{}
                     )
    module.hot.accept('./js/sum')
}

      4. oneof ， include/exclude

      5. Cache 对 Eslint ，Babel编译的文件进行缓存 第二次打包时就不用所有文件都要通过babel处理，只有改变了个文件要Eslint ，Bael重新处理

      6. Thead 开启多进程打包 我们前面提到过在生产模式下 打包好的 js文件会被默认压缩，其实是 Terser插件在处理，js压缩

      7. 开启多进程打包

        1. 先引入 os模块 通过 os.cpus().length()拿到cpu的核数

        2. 下载 npm i thread-loader -D

        3. 对 Eslint Babel Terser开启多进程

        4. 例子：

      # 优化代码运行性能

      1. 模块动态导入 开启 chunks:all ,使用import自动导入

      2. 给动态导入的代码命名，给chunk命名

      3. preload/prefetch 让浏览器空闲的时候加载需要懒加载或公共模块的chunk，避免了当需要加载的时候资源过大，用户体验不好。对chunk文件 做 preload引入

        1. preload: 告诉浏览器立即加载资源，只能加载当前页面的资源
        2. prefetch：空闲时在加载，既可以加载当前页面的资源，也可以加载下一个需要使用的资源
        3. 共同点:都会加载，但不执行，都会有缓存

      4. Network-cache ,

        1. chunk文件更新 引入chunk的文件不用更新，使得能更好的使用缓存 加快打包速度，
        2. 使用 contenthash 输出的文件根据文件内容生成hash值文件不变hash值不变。
        3. 增加 runtimeChunk配置 生成 runtime时文件 ，runtime文件保存有文件之间依赖的hash值，当chunk文件变化时，只有chunk 和runtime变化，使得引用chunk的文件不受影响。

        2. 使用 Core-js专门做一些Es6以上的API兼容性处理，通过打补丁的形式。
